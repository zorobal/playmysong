## 1. Problèmes Rencontrés

### 1.1 Problèmes Techniques Identifiés

| # | Problème | Statut | Impact |
|---|----------|--------|--------|
| 1 | **CRUD Playlists incomplet** - Les playlists créées ne s'affichent pas toujours après création | Non résolu | Moyen |
| 2 | **File d'attente persistante** - Les chansons jouées restent affichées | Partiellement résolu | Moyen |
| 3 | **Playlist Établissement** - Ne joue pas correctement en boucle | Partiellement résolu | Moyen |
| 4 | **Synchronisation temps réel** - Les pages ne se mettent pas à jour en temps réel | Partiellement résolu | Élevé |
| 5 | **URL QR Code visible** - L'URL peut être copiée/partagée | Non résolu | Élevé |
| 6 | **Vercel + Socket.IO** - WebSocket ne fonctionne pas sur Vercel (serverless) | Connu, limitation | Élevé |
| 7 | **Cache Frontend** - Les fichiers JS anciens sont servis | Récurrent | Faible |

### 1.2 Problèmes de Logique Métier

| # | Problème | Statut | Impact |
|---|----------|--------|--------|
| 1 | **Double demande** - Un client peut faire plusieurs demandes sans rescanner | Résolu | Moyen |
| 2 | **YouTube API non configurée** - Recherche YouTube indisponible | Bloquant | Élevé |
| 3 | **No NowPlaying auto-play** - Les chansons ne se jouent pas automatiquement | Non résolu | Élevé |
| 4 | **Pas de fallback** - Si playlist établissement est vide, rien ne joue | Non résolu | Moyen |

### 1.3 Problèmes d'Infrastructure

| # | Problème | Statut | Impact |
|---|----------|--------|--------|
| 1 | **Vercel Serverless** - WebSocket non supporté, polling uniquement | Connu | Élevé |
| 2 | **Base de données Neon** - Connexion parfois lente | Connu | Faible |
| 3 | **Firebase Hosting** - Cache agressif, hard refresh nécessaire | Récurrent | Moyen |

### 1.4 Solutions Temporaires Appliquées

1. **Suppression du polling automatique** - Remplacé par mise à jour locale immédiate + bouton refresh
2. **Route publique playlists** - `/api/playlists/by-establishment/:id` accessible sans auth
3. **Socket.IO-events locaux** - Émission d'événements sans réception (Vercel limite)

## 2. Description du Projet

### 2.1 Présentation Générale

**PlayMySong** est une application web/mobile permettant aux clients d'établissements (bars, restaurants, discothèques) de demander des chansons qui seront jouées dans l'établissement.

### 2.2 Architecture Globale

```
┌─────────────────────────────────────────────────────────────────┐
│                         CLIENTS (PWA Mobile)                     │
│    QR Code → Page Client → Recherche/Playlist → Soumettre       │
└─────────────────────────────────────────────────────────────────┘
                                  │
                                  ▼
┌─────────────────────────────────────────────────────────────────┐
│                         BACKEND (Vercel)                         │
│                    API REST + Socket.IO (polling)                │
└─────────────────────────────────────────────────────────────────┘
                                  │
                                  ▼
┌─────────────────────────────────────────────────────────────────┐
│                    BASE DE DONNÉES (Neon PostgreSQL)             │
│         Establishments │ Users │ Playlists │ SongRequests        │
└─────────────────────────────────────────────────────────────────┘
                                  │
                                  ▼
┌─────────────────────────────────────────────────────────────────┐
│              DASHBOARDS (Admin/Operator/User)                     │
│          Validation des demandes │ Gestion playlists            │
└─────────────────────────────────────────────────────────────────┘
                                  │
                                  ▼
┌─────────────────────────────────────────────────────────────────┐
│                    ÉCRAN GÉANT (Électron)                       │
│              Affichage NowPlaying + File d'attente                │
└─────────────────────────────────────────────────────────────────┘
```

## 4. Cahier des Charges

### 4.1 Objectifs du Projet

#### Objectif Principal
Permettre aux clients d'un établissement de demander des chansons via QR Code, avec validation par le personnel et affichage sur un écran géant.

#### Objectifs Secondaires
1. Gestion multi-établissements
2. Playlists personnalisables
3. Mode "Playlist Établissement" (fallback)
4. Interface responsive mobile-first
5. PWA installable

### 4.2 Utilisateurs et Rôles

| Rôle | Droits | Interface |
|------|--------|-----------|
| **CLIENT** | Scanner QR, rechercher/choisir chanson, ajouter message/selfie | PWA Mobile |
| **OPERATOR** | Voir demandes, valider/rejeter | Dashboard Operator |
| **ADMIN** | Operator + gérer playlists, utilisateurs | Dashboard Admin |
| **SUPER_ADMIN** | Admin + gérer établissements | Dashboard SuperAdmin |

### 4.3 Fonctionnalités Détaillées

#### 4.3.1 Flux Client (PWA Mobile)

```
1. Client scanne QR Code
   ↓
2. Redirection vers /client?establishmentId=XXX
   ↓
3. Client voit deux onglets:
   - 📁 Playlist (chansons de l'établissement)
   - 🔍 YouTube (recherche, si API configurée)
   ↓
4. Client sélectionne une chanson
   ↓
5. Optionnel: Message texte + Selfie
   ↓
6. Soumission de la demande
   ↓
7. Confirmation + option "Nouvelle demande" → retour à l'accueil
```

**Contraintes:**
- Une seule demande par scan QR Code
- Pas d'authentification requise pour le client
- Fonctionne hors-ligne après premier chargement (PWA)

#### 4.3.2 Flux Admin/Operator

```
1. Connexion avec email/mot de passe
   ↓
2. Tableau de bord avec 5 onglets:
   - 📩 Demandes (PENDING)
   - 📁 Playlist (CRUD playlists)
   - 🎼 Playlist Établissement
   - 👥 Utilisateurs
   - 📊 Statistiques
   ↓
3. Validation d'une demande:
   - Clic sur "✅ Valider"
   - La chanson apparaît dans la file d'attente
   ↓
4. Rejet d'une demande:
   - Clic sur "❌ Rejeter"
   - Saisie optionnelle du motif
   - La demande est supprimée
```

#### 4.3.3 Flux NowPlaying (Écran Géant)

```
1. Affichage en plein écran (1920x1080)
   ↓
2. Vérification de la file d'attente:
   
   FILE D'ATTENTE VIDE?
   ├── OUI → Joue Playlist Établissement en boucle
   │         └── Si Playlist Établissement vide → Écran d'attente
   │
   └── NON → Joue file d'attente (priorité haute)
             └── Fin de chanson → Marquer COMPLETED → Prochaine chanson
   ↓
3. Affichage:
   - Titre + Artiste
   - Miniature YouTube
   - Selfie client (si disponible)
   - Message client (si disponible)
   - Badge "Demande" ou "Playlist Établissement"
   ↓
4. File d'attente visible en bas de l'écran
```

#### 4.3.4 Gestion des Établissements (SuperAdmin)

```
1. Liste des établissements
   ↓
2. Pour chaque établissement:
   - Nom, Ville, Quartier
   - QR Code (généré automatiquement)
   - Utilisateurs liés
   - Playlists
   ↓
3. Actions:
   - Créer nouvel établissement
   - Modifier
   - Supprimer (avec cascade)
   - Ajouter des admins
```

### 4.4 Exigences Non Fonctionnelles

| Critère | Exigence |
|---------|----------|
| **Performance** | Temps de réponse < 200ms pour les API |
| **Disponibilité** | 99.5% uptime |
| **Sécurité** | JWT avec expiration 7j, hash bcrypt |
| **Compatibilité** | Chrome, Firefox, Safari, Edge (versions récentes) |
| **Mobile** | iOS Safari, Chrome Android |
| **Accessibilité** | Contraste suffisant, navigation clavier |

### 4.5 Contraintes Techniques

| Contrainte | Description |
|------------|-------------|
| **Vercel Serverless** | WebSocket non supporté, utiliser polling ou mises à jour locales |
| **Neon PostgreSQL** | Connexions limitées, utiliser connection pool |
| **YouTube API** | Quota limité, nécessitant clé API |
| **Firebase Hosting** | Cache agressif, utiliser versioning |

### 4.6 Cas Limites à Gérer

1. **File d'attente vide** → Jouer Playlist Établissement
2. **Playlist Établissement vide** → Écran d'attente
3. **Aucune chanson jouée depuis 24h** → Alerte Admin
4. **Client sans selfie/message** → Affichage par défaut
5. **Playlist supprimée alors qu'elle est "Établissement"** → Fallback vers autre playlist
6. **Établissement inactif** → Désactiver QR Code

### 5.3 Modèle de Données (Prisma)

```prisma
model User {
  id              String   @id @default(cuid())
  email           String   @unique
  password        String
  role            Role     @default(OPERATOR)
  name            String?
  phoneNumber     String?
  establishmentId String?
  createdBy       String?
  isActive        Boolean  @default(true)
  establishment   Establishment? @relation(...)
}

model Establishment {
  id              String   @id @default(cuid())
  name            String
  city            String?
  district        String?
  phoneNumber     String?
  additionalInfo  String?
  playlists       Playlist[]
  users           User[]
  songRequests    SongRequest[]
}

model Playlist {
  id                    String   @id @default(cuid())
  establishment         Establishment @relation(...)
  establishmentId       String
  name                  String
  createdBy             String?
  isEstablishmentPlaylist Boolean @default(false)
  songs                 Song[]
}

model Song {
  id         String   @id @default(cuid())
  playlist   Playlist @relation(...)
  playlistId String
  title      String
  artist     String?
  filePath   String?
  youtubeId  String?
  thumbnail  String?
  duration   Int      @default(0)
}

model SongRequest {
  id                String   @id @default(cuid())
  establishment     Establishment @relation(...)
  establishmentId  String
  youtubeId         String?
  title             String
  artist            String?
  thumbnail         String?
  durationSec       Int      @default(0)
  message           String?
  selfieUrl         String?
  status            RequestStatus @default(PENDING)
  rejectionReason   String?
  createdAt         DateTime @default(now())
}

enum Role {
  SUPER_ADMIN
  ADMIN
  OPERATOR
  USER
}

enum RequestStatus {
  PENDING
  VALIDATED
  REJECTED
  PLAYING
  COMPLETED
}
```

### 5.5 Socket.IO Events

**Note:** Les WebSockets ne fonctionnent pas sur Vercel Serverless. Les événements sont émis mais non reçus côté client (sauf polling).

| Event | Direction | Payload | Usage |
|-------|-----------|---------|-------|
| `join_establishment` | Client → Server | `{ establishmentId }` | Rejoindre room |
| `leave_establishment` | Client → Server | `{ establishmentId }` | Quitter room |
| `request_validated` | Server → Client | `{ requestId, request }` | Demande validée |
| `request_rejected` | Server → Client | `{ requestId, request }` | Demande rejetée |
| `now_playing_updated` | Server → Client | `{ request }` | Chanson en cours |
| `request_completed` | Server → Client | `{ requestId }` | Chanson terminée |

### 6.2 Points d'Attention pour l'Équipe

#### ⚠️ Limitations Connues

1. **Vercel Serverless**
   - Socket.IO fonctionne en mode polling uniquement
   - Les événements WebSocket ne sont pas fiables
   - Solution: Mise à jour locale + bouton refresh

2. **YouTube API**
   - Non configurée dans l'environnement actuel
   - Ajouter `YOUTUBE_API_KEY` dans `.env`
   - Alternative: Ajouter songs manuellement via Admin

3. **Playlist Établissement**
   - Marquer une playlist comme "Établissement" via Admin
   - Sert de fallback quand queue vide
   - Joue en boucle

4. **Cache Frontend**
   - Après déploiement, faire Ctrl+Shift+R (hard refresh)
   - Service Worker peut servir d'anciens fichiers

#### 🎯 Priorités de Développement

1. **Corriger auto-play** - Les chansons doivent se jouer automatiquement
2. **Implémenter temps réel** - Synchroniser les dashboards
3. **Configurer YouTube API** - Recherche fonctionnelle
4. **Tests** - Couverture minimum 70%
